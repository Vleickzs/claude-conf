## Description

Le supervisor-guard bloque Write et Edit sur du code source, mais ne monitore pas le tool Bash. Un supervisor peut contourner le guard avec `echo "code" > file.ts`, `sed -i 's/old/new/' file.ts`, `tee file.ts`, ou un heredoc `cat <<EOF > file.ts`.

En pratique, ce contournement est moins probable qu'un Edit direct (le LLM prefere les outils natifs), mais c'est un trou dans l'enforcement mecanique.

## Approche recommandee

Ajouter un matcher **Bash** au supervisor-guard hook :

1. Quand Bash est invoque en mode supervisor, extraire la commande depuis `tool_input.command`
2. Scanner la commande pour des patterns d'ecriture fichier :
   - Redirections : `>`, `>>` suivis d'un path hors whitelist
   - Commandes : `tee`, `sed -i`, `awk -i inplace`, `cp ... target`, `mv ... target`
   - Heredocs : `cat <<` suivi d'un `>` vers un fichier
3. Si la cible est hors whitelist â†’ bloquer avec le meme format JSON que Write/Edit

**Pattern similaire a command-guard** qui filtre deja les commandes dangereuses (rm -rf, etc.). Peut reutiliser l'approche regex de command-guard.

**Risque de faux positifs** : les commandes comme `git diff > /tmp/output` ou `ls > listing.txt` pourraient etre bloquees a tort. Mitigation : ne bloquer que si le fichier cible a une extension de code connue (.ts, .js, .sh, .py, .dart, .go, .rs, .swift, etc.) OU est dans un repertoire source (src/, lib/, bin/, hooks/).
