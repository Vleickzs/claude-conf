# IMP-024: supervisor — analyser les problèmes découverts AVANT le verdict global

**Type:** Improvement
**Statut:** A faire
**Priorite:** Haute
**Complexite:** XS
**Tags:** supervisor
**Date creation:** 2026-03-21

---

## Description

La section 5bis (analyse des problèmes découverts) est bien écrite mais positionnée APRÈS le verdict global dans le flow du supervisor. En pratique, le supervisor fait son évaluation, conclut "tout est bon", puis traite les problèmes comme un appendice optionnel. Les problèmes sont minimisés par omission ou regroupés sous un verdict vague.

## Cause racine

Ordre de traitement dans le prompt : le supervisor évalue la qualité (étape 5), donne un verdict (OK/KO), PUIS analyse les problèmes (5bis). L'ancrage cognitif est déjà fait — le verdict est pris avant l'analyse des problèmes.

## Approche

Trois modifications dans `supervisor/commands/supervisor.md` :

### 1. Réordonner : problèmes AVANT verdict

Déplacer l'analyse des problèmes découverts (actuelle section 5bis) AVANT le bloc "If OK / If NOT OK". Le flow devient :

```
1. Read the report
2. Check modified files
3. Run checks
4. Check the diff (scopé)
5. Evaluate quality
6. **Analyze discovered problems** (ex-5bis) — AVANT le verdict
7. **Present problems to user with individual verdicts**
8. THEN give global verdict (OK/KO) — qui intègre l'analyse des problèmes
```

### 2. Ajouter instruction explicite

Après l'analyse des problèmes, avant le verdict :
```
**Tu présentes les problèmes découverts à l'utilisateur AVANT de donner ton verdict.
Le verdict global intègre ton évaluation des problèmes — pas l'inverse.
Si un problème découvert est un bug ou un risque, le verdict NE PEUT PAS être "OK"
sans que le problème ait un ticket créé.**
```

### 3. Règle anti-regroupement

Ajouter dans la section problèmes :
```
**Chaque problème = 1 ligne dans le tableau avec son propre verdict et sa propre action.
Jamais de regroupement ("points mineurs"), jamais de verdict collectif.**
```
