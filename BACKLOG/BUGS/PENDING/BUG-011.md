# BUG-011: supervisor-guard — activation fragile, depend d'un touch dans le prompt

**Type:** Bug
**Statut:** A faire
**Priorite:** Haute
**Complexite:** M
**Tags:** supervisor-guard, hooks, fiabilite
**Depends on:** none
**Blocked by:** —
**Date creation:** 2026-03-30

---

## Description

Le supervisor-guard (PreToolUse hook) ne s'active que si un marker file `.claude-sessions/supervisor-active/{session_id}` existe. Ce marker est cree par une instruction dans le prompt supervisor (`touch .claude-sessions/supervisor-active/$SESSION_ID`). Si le LLM skip cette etape (contexte long, urgence, drift), le guard est completement inactif — Write/Edit passent sans blocage.

Incident reel : un supervisor a directement edite du code source (Edit sur un .dart) sans que le guard ne bloque, probablement parce que le marker n'a jamais ete cree.

## Cause racine

L'activation repose sur une instruction textuelle, pas sur un mecanisme mecanique. C'est l'equivalent d'un verrou dont la cle est "penser a fermer la porte".

## Approche recommandee

Ajouter un hook **UserPromptSubmit** qui detecte automatiquement le mode supervisor et cree le marker sans intervention du LLM :

1. Le hook intercepte le premier message utilisateur
2. Il scanne le contenu pour detecter la presence du pattern supervisor (ex: `<command-name>/supervisor</command-name>` ou le tag `SUPERVISOR MODE` dans le prompt)
3. S'il detecte le mode supervisor → `mkdir -p .claude-sessions/supervisor-active && touch .claude-sessions/supervisor-active/{session_id}`
4. Le guard PreToolUse existant prend le relai normalement

**Alternative** : pour les supervisors lances via script (`.claude-sessions/launch/`), le launch script peut setter un env var `CC_SUPERVISOR_SESSION=1`. Le hook PreToolUse check cette var EN PLUS du marker file. Double filet.

**Alternative 2** : inverser la logique — le hook bloque Write/Edit par defaut si `.claude-sessions/supervisor-active/` existe comme directory (meme vide), et c'est le mode WORKER qui cree un marker d'exemption. Plus strict mais change le contrat.

## Fichiers concernes

- `supervisor-guard/hooks/supervisor-guard.sh` — ajouter fallback detection (env var check)
- `supervisor-guard/hooks/supervisor-detect.sh` — nouveau hook UserPromptSubmit (auto-detection)
- `supervisor-guard/install.sh` — installer le nouveau hook + configurer le matcher UserPromptSubmit
- `supervisor/commands/supervisor.md` — le touch explicite peut rester comme backup, mais n'est plus le seul mecanisme

## Criteres d'acceptation

- [ ] En mode supervisor lance via `/supervisor`, le guard bloque Write/Edit sur du code source SANS que le LLM ait besoin de creer manuellement le marker
- [ ] En mode supervisor lance via script, le guard est actif des le premier tool call
- [ ] En mode normal (pas de supervisor), aucun blocage (pas de regression)
- [ ] Le marker est toujours nettoye en fin de session supervisor
- [ ] Le mecanisme existant (touch explicite) continue de fonctionner comme fallback
